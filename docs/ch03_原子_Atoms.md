## 3.0 设计意图：不可变字符串与字符串驻留

**Atom**（原子）是一个指向**唯一的、不可变的、零个或多个任意字节序列**的指针。大多数原子是指向以 null 结尾的字符串的指针，但指向任何字节序列的指针都可以成为原子。

其核心思想来源于 LISP 语言中的 "atom" 概念，在现代编程语言中通常被称为**字符串驻留 (string interning)**——每个唯一的字符串在内存中仅存在一份副本，所有内容相同的字符串指针指向同一块内存。

### 为什么需要 Atom？

1.  **指针相等即内容相等**：比较两个字节序列是否相等，只需比较它们的指针。这是原子的最大优势——将 O(n) 的字符串比较降为 O(1) 的指针比较。
2.  **节省空间**：对于重复出现的相同字符串，内存中只存在一份副本。
3.  **作为复合键使用**：原子常用作以任意字节序列为索引的数据结构（如表 Table、集合 Set，见第 8、9 章）中的键。

### 核心约束

*   **检查运行时错误**：传给任何接口函数的指针不能为 `NULL`；`Atom_new` 的 `len` 不能为负数；`Atom_length` 只能接收一个原子的指针（否则 `assert(0)` 触发）。
*   **未检查运行时错误**：修改原子指向的字节内容是未定义行为——原子是不可变的。
*   **生命周期**：原子一旦创建，在客户端程序的整个生命周期中存在，永不释放。
*   **异常**：`Atom_new`、`Atom_string` 和 `Atom_int` 可能在内存不足时抛出 `Mem_Failed` 异常。

## 3.2 哈希表数据结构

### 3.2.1 顶层结构 — buckets 数组

原子表是一个采用**链地址法 (separate chaining)** 的哈希表：

```c
/* NELEMS — 计算编译时数组元素个数的经典 C 语言宏。
 * 示例: NELEMS(buckets) = sizeof(buckets) / sizeof(buckets[0]) = 2048 */
#define NELEMS(x) ((sizeof (x))/(sizeof ((x)[0])))

/* buckets 数组 — 哈希表的顶层，固定 2048 个槽位 (2^11)。
 * 每个桶是一个单向链表的头指针。
 * 使用 2 的幂而非素数是作者的有意选择 — 见练习 3.1 的讨论。 */
static struct atom {
    struct atom *link;   /* 指向链表中下一个条目 (NULL 表示末尾) */
    int len;             /* 该原子的字节序列长度 (不含 '\0') */
    char *str;           /* 指向字节序列（实际存储于 (p+1) 处） */
} *buckets[2048];
```

### 3.2.2 数据结构示意图

```
buckets[0]    -> atom_A -> atom_B -> NULL
buckets[1]    -> NULL
buckets[2]    -> atom_C -> NULL
    ...
buckets[h]    -> atom_X -> atom_Y -> atom_Z -> NULL
    ...
buckets[2047] -> atom_W -> NULL
```

*   `buckets[i]` 指向的链表包含所有哈希值 mod 2048 后等于 `i` 的原子。
*   新条目总是插入到链表的**头部**（头插法）——在头部插入只需 O(1) 时间，且热点数据自动前置。

### 3.2.3 变长结构体 — 内存布局

每个 `struct atom` 仅包含其特定序列所需的空间。对于 `Atom_string("an atom")`（长度 7），在小端序 32 位机器上的内存布局：

```
                  ALLOC(sizeof(*p) + 7 + 1) 分配的一块连续内存
                  |<- sizeof(*p) = 12 ->|<-- 7 bytes -->|<- 1B ->|
                  +---------------------+----------------+--------+
偏移量(小端序):    | link(4B)            | 'a'            | 'o'    |
                  | len=7(4B)           | 'n'            | 'm'    |
                  | str(4B) ----------> | ' '            | '\0'   |
                  +---------------------+----------------+--------+
                                        ^
                                        p->str = (char *)(p + 1)
```

关键设计：`p->str` 并非指向另一块独立分配的内存，而是在**同一块连续内存**中跟在结构体之后。这避免了两层间接寻址，提升了缓存局部性。

### 3.2.4 练习 3.4 的替代方案 — C99 柔性数组成员

原书练习 3.4 讨论了另一种声明方式：

```c
/* 练习 3.4 中的替代声明 — 使用长度为 1 的数组模拟变长结构体 */
struct atom {
    struct atom *link;
    int len;
    char str[1];     /* 原书中的 "trick" — 违反 C89 标准 */
};

/* 分配: ALLOC(sizeof(*p) + len) — 不需要额外的 +1，因为 str[1] 已包含 1 字节 */
/* 在现代 C (C99+) 中应使用柔性数组成员 (Flexible Array Member):
   char str[];  — 标准且安全 */
```

原书作者指出此 "trick" 违反 C 标准（访问 `str[0]` 之外的字节是未定义行为），但实践中被广泛使用。

## 3.3 哈希算法分析

### 3.3.1 scatter 查表法

`Atom_new` 使用一个 256 个条目的 `unsigned long` 数组 `scatter`，将每个字节值 (0-255) 映射到一个伪随机数：

```c
/* scatter 数组 — 字节→随机数映射表，用于哈希计算。
 * 256 个 unsigned long 条目，由标准库函数 rand() 预生成。
 * 将单个字节映射为伪随机大数，改善哈希分布均匀性。 */
static unsigned long scatter[] = {
2078917053, 143302914, 1027100827, 1953210302, 755253631, 2002600785,
1405390230, 45248011, 1099951567, 433832350, 2018585307, 438263339,
/* ... 共 256 个条目 (已省略中间的条目) ... */
1884137923, 53392249, 1735424165, 1602280572
};
```

### 3.3.2 哈希计算过程

```c
/* h ← hash str[0..len-1] — 书本公式
 *
 * 算法: 对序列的每个字节，执行 h = (h << 1) + scatter[byte]
 * - (h << 1) 相当于 h * 2，使每个字节的贡献具有位置权重（类似于多项式哈希）
 * - scatter[(unsigned char)str[i]] 将字节映射为伪随机大数后累加
 * - (unsigned char) 强制转换避免符号扩展：
 *   在 signed char 机器上，值 > 127 会产生负索引，导致数组越界
 */
for (h = 0, i = 0; i < len; i++)
    h = (h << 1) + scatter[(unsigned char)str[i]];

/* 将哈希值映射到桶索引 [0, 2047]。
 * h &= 2047 等价于 h % 2048，利用 2 的幂特性用按位与替代模运算。
 * 原书中使用: h %= NELEMS(buckets);
 * 实际代码使用: h &= NELEMS(buckets)-1;  (更快) */
h &= NELEMS(buckets) - 1;   /* NELEMS(buckets)-1 = 2047 = 0b111_1111_1111 */
```

### 3.3.3 为什么需要 scatter？

*   **直接使用 ASCII 值的缺陷**：英文文本中字符分布高度不均匀（字母集中在 65-122 范围内），直接代入哈希函数会导致某些桶的链长远远超过其他桶。
*   **scatter 的作用**：将每个字节值 "喷洒" 到整个 32 位空间，消除了输入数据的统计偏差。

### 3.3.4 算法特性

| 特性 | 分析 |
|------|------|
| **均匀性** | scatter 表使用 `rand()` 生成，提供了良好的字节级随机性 |
| **冲突率** | 当 N（原子数）< 2 x NELEMS(buckets) = 4096 时，平均链长 < 1，查找接近 O(1) |
| **桶数选择** | 2048 = 2^11，属 2 的幂而非素数。作者在练习 3.1 中验证了这一选择在配合 scatter 时的效果 |
| **字符类型处理** | `(unsigned char)str[i]` 是关键——避免了在 signed char 平台上值 > 127 导致的负索引 |
| **哈希函数来源** | 由 Hans Boehm 建议，也用于 lcc 编译器 (Fraser and Hanson 1995) |

### 3.3.5 为什么不使用 `strncmp`（练习 3.3）

`Atom_new` 不使用标准库函数 `strncmp` 来比较序列，原因有二：

1.  `strncmp` 比较的是以 null 结尾的字符串。原子可以是任意字节序列，可能包含内嵌的 `'\0'` 字符。
2.  先比较 `len` 字段可以在大多数情况下快速排除不匹配（长度不同的序列必然不同），避免了不必要的逐字节比较。

```c
/* ============ Atom_int — 将 long 整数转为十进制字符串后创建原子 ============ */
/* 核心挑战:
 *   1. 补码系统的非对称范围 — LONG_MIN 的绝对值无法用 long 表示
 *   2. C 语言有符号除法和取模的歧义性
 * 解决方案: 全部使用无符号 (unsigned long) 算术 */

const char *Atom_int(long n) {
    char str[43];                      /* 43 = ceil(128/3) + 1 字符:
                                        * 任意 128-bit 整数的八进制表示最多 128/3+1 = 43 位,
                                        * 十进制位数 ≤ 八进制位数, 故 43 字符始终足够 */
    char *s = str + sizeof str;        /* s 指向缓冲区末尾之后的第一个字节 (str+43),
                                        * 将从右向左递减地填充数字字符 */
    unsigned long m;                   /* 使用 unsigned long 避免有符号运算的未定义行为 */

    /* —— 处理 LONG_MIN: 在补码系统中 -LONG_MIN 会溢出 ——
     * LONG_MIN 通常为 -2^63, LONG_MAX 为 2^63-1
     * LONG_MAX + 1UL = 2^63, 恰好等于 |LONG_MIN| (在 unsigned long 中精确可表示) */
    if (n == LONG_MIN)                 /* 如果 n 是最负的 long 值 (特殊情况) */
        m = LONG_MAX + 1UL;            /* 用 unsigned long 表示其绝对值: 2^63 */
    else if (n < 0)                     /* n 为负数但不是 LONG_MIN */
        m = -n;                         /* 安全地取绝对值 */
    else                                /* n >= 0 */
        m = n;                          /* 直接赋值, 无需转换 */

    /* —— 从右向左生成十进制数字 ——
     * 使用 do...while 而非 while 确保至少执行一次 (处理 n == 0 的情况) */
    do
        *--s = m % 10 + '0';            /* 取 m 的个位数 (0-9) → 转为 ASCII 字符,
                                         * 存入 --s (指针先递减再写入) */
    while ((m /= 10) > 0);              /* m = m / 10 (去掉个位), 若 m > 0 则继续 */

    /* —— 如果原值为负, 在前方添加负号 —— */
    if (n < 0)
        *--s = '-';                      /* s 递减后写入 '-' */

    /* s 现在指向转换后的字符串的第一个字符。
     * 字符串长度为 (str + sizeof str) - s, 即从 s 到缓冲区末尾的字符数 */
    return Atom_new(s, (str + sizeof str) - s);
}


/* ============ Atom_new — 原子表的核心函数: 查找或创建原子 ============ */
/* 流程:
 *   1. 计算哈希值 (scatter 查表 + 左移)
 *   2. 在对应桶的链表中查找已有原子
 *   3. 找到 → 返回已有原子的 str 指针
 *   4. 未找到 → 分配新条目, 头插到链表, 返回新原子的 str 指针 */

const char *Atom_new(const char *str, int len) {
    unsigned long h;                     /* 哈希值 (unsigned long 类型) */
    int i;                               /* 通用循环计数器 */
    struct atom *p;                      /* 链表遍历指针 */

    assert(str);                         /* 检查运行时错误: str 不能为 NULL */
    assert(len >= 0);                    /* 检查运行时错误: len 不能为负数 */

    /* ======== 步骤 1: 计算哈希值 ========
     * 算法: h_0 = 0; h_{i+1} = (h_i << 1) + scatter[str[i]]
     * (h << 1) 相当于 h * 2, 使每个字节的贡献具有位置权重
     * scatter[(unsigned char)str[i]] 将字节值映射为伪随机大数 */
    for (h = 0, i = 0; i < len; i++)
        h = (h << 1) + scatter[(unsigned char)str[i]];

    /* 将哈希值映射到桶索引 [0, 2047]
     * h &= 2047 等价于 h % 2048, 利用 2 的幂加速 (按位与比取模快得多) */
    h &= NELEMS(buckets) - 1;         /* NELEMS(buckets)-1 = 2047 = 0b111'1111'1111 */

    /* ======== 步骤 2: 在 buckets[h] 链表中查找已有原子 ======== */
    for (p = buckets[h]; p; p = p->link) {  /* 遍历链表, p == NULL 时停止 */
        /* 快速路径: 先比较长度, 不等则跳过 (避免不必要的逐字节比较) */
        if (len == p->len) {
            /* 长度匹配 → 逐字节比较内容
             * 循环体为空 — 所有比较工作全在 for 的条件部分完成 */
            for (i = 0; i < len && p->str[i] == str[i]; )
                i++;
            /* 如果 i == len, 说明所有 len 个字节全部匹配 → 找到已有原子 */
            if (i == len)
                return p->str;           /* 返回已存在原子的 str 指针 */
        }
    }

    /* ======== 步骤 3: 未找到 → 分配新条目 ========
     * 分配空间: sizeof(struct atom) + len + 1 字节
     *   - sizeof(*p): 结构体本身 (link + len + str 指针, 共 3 个字)
     *   - len:        字节序列内容
     *   - +1:         末尾的 null 终止符 '\0'
     * ALLOC 是 Mem_alloc 的宏, 失败时抛出 Mem_Failed 异常 */
    p = ALLOC(sizeof (*p) + len + 1);
    p->len = len;                        /* 记录序列长度 */
    /* str 字段指向结构体之后紧跟的内存区域。
     * (p + 1) 的指针算术: p 是 struct atom *, p+1 跳过 sizeof(*p) 字节 */
    p->str = (char *)(p + 1);            /* 变长结构体的核心技巧 */
    if (len > 0)
        memcpy(p->str, str, len);        /* 复制字节序列 (len == 0 时跳过, 避免调用 memcpy 处理空序列) */
    p->str[len] = '\0';                  /* 强制写入 null 终止符, 确保与 C 字符串兼容 */

    /* —— 头插法: 将新条目插入 buckets[h] 链表头部 ——
     * 在头部插入只需 O(1) 时间, 且热点数据自动前置 */
    p->link = buckets[h];                /* 新条目的 link 指向旧的链表头 */
    buckets[h] = p;                      /* 桶头指针指向新条目 */

    return p->str;                       /* 返回新原子的 str 指针 */
}


/* ============ Atom_length — 查找并返回给定原子的长度 ============ */
/* 由于调用者只提供了 str 指针 (没有长度信息), 无法计算哈希值,
 * 必须在所有桶的所有链表中进行线性扫描。
 * 时间复杂度: O(已创建的原子总数), 这是该接口的设计权衡。 */

int Atom_length(const char *str) {
    struct atom *p;                      /* 链表遍历指针 */
    int i;                               /* 桶索引 (循环变量) */

    assert(str);                         /* 检查运行时错误: str 不能为 NULL */

    /* 双层嵌套循环: 遍历所有 2048 个桶的外层 + 遍历每个链表的内层 */
    for (i = 0; i < NELEMS(buckets); i++)
        for (p = buckets[i]; p; p = p->link)
            /* 关键: 比较指针本身 (p->str == str), 而非字符串内容!
             * 因为原子指针是全局唯一的, 内容相同的字符串必然指针相同。 */
            if (p->str == str)
                return p->len;           /* 找到 → 返回该原子的长度 */
    /* 如果执行到这里, 说明 str 不是任何已知原子的指针。
     * assert(0) 触发 "can't happen" 断言 — 实现检查运行时错误。 */
    assert(0);
    return 0;                            /* 仅在 NDEBUG 编译时到达此处 (assert 被禁用时) */
}
```

## 3.5 Atom_int 中的整数转字符串算法详解

`Atom_int` 是本章中最精巧的函数之一。它面临三个关键挑战：

### 3.5.1 挑战一：补码系统的不对称性

在 C 语言的补码系统中：

```
LONG_MIN = -2^63  = -9,223,372,036,854,775,808
LONG_MAX =  2^63-1 = 9,223,372,036,854,775,807
|LONG_MIN| = 2^63 > LONG_MAX  （溢出！）
```

如果直接写 `m = -n;` 而当 `n == LONG_MIN` 时，结果是**未定义行为**。解决：

```c
if (n == LONG_MIN)
    m = LONG_MAX + 1UL;   /* unsigned long: 2^63, 恰好等于 |LONG_MIN| */
```

这里 `LONG_MAX + 1UL` 是合法且精确的——在 unsigned long 中，它等 `2^63`。

### 3.5.2 挑战二：C 语言有符号除法和取模的歧义性

C89 标准规定有符号整数除法 `a / b` 向零截断还是向负无穷截断是**实现定义的 (implementation-defined)**。同理，`%` 运算符的结果符号也是实现定义的。使用无符号算术完全避免了这些歧义。

### 3.5.3 挑战三：从右向左生成

```c
char str[43];
char *s = str + sizeof str;     /* 指向缓冲区末尾 */
do
    *--s = m % 10 + '0';        /* 从右向左逐位填充 */
while ((m /= 10) > 0);
```

*   **为何从右向左？** 数字的十进制表示需要从最低位开始计算。如果从左向右写入，需要提前知道数字的总位数，或者在末尾反转字符串。
*   **为何 `do...while`？** 保证即使 `m == 0`（输入为 0）也至少执行一次循环，生成单字符 `"0"`。
*   **缓冲区为何是 43？** 推理链：128 位有符号整数的八进制最多 `128/3 + 1 = 43` 位 → 十进制位数不超过八进制位数 → 43 足够容纳任何可能的 long。即使未来硬件扩展到 128 位 long，代码无需修改。

## 3.6 内存管理：ALLOC 宏与引用语义

### 3.6.1 ALLOC 宏的来源（来自 `mem.h`）

```c
/* mem.h — CII 的内存管理接口 */
#include "except.h"                    /* 异常处理框架 */

extern const Except_T Mem_Failed;      /* 内存分配失败时抛出的异常 */

/* 底层分配函数: 封装 malloc/calloc/free/realloc, 失败时抛出异常 */
extern void *Mem_alloc (long nbytes, const char *file, int line);
extern void *Mem_calloc(long count, long nbytes, const char *file, int line);
extern void  Mem_free  (void *ptr, const char *file, int line);
extern void *Mem_resize(void *ptr, long nbytes, const char *file, int line);

/* ALLOC 宏 — 将 __FILE__ 和 __LINE__ 自动注入, 便于错误定位 */
#define ALLOC(nbytes) \
    Mem_alloc((nbytes), __FILE__, __LINE__)
```

### 3.6.2 为什么 Atom_new 用 ALLOC 而不是 NEW？

`Mem` 模块提供两种分配宏：

*   `NEW(p)` — `((p) = ALLOC((long)sizeof *(p)))`，适用于**编译时已知大小**的场景。
*   `ALLOC(nbytes)` — 适用于**运行时才知道大小**的场景。

`Atom_new` 中 `sizeof(*p) + len + 1` 的大小依赖运行时的 `len` 参数，因此必须使用 `ALLOC`。

### 3.6.3 原子的生命周期 — 永不释放

*   **设计决策**：原子一旦创建，在程序运行期间**永远不会被释放**。
*   **实践经验支撑**：这一设计来源于 `lcc` 编译器 (Fraser and Hanson 1995) —— C 程序中不同字符串的数量惊人地少，不论源程序多大，永不释放也从不消耗过多存储。
*   **后果**：简化了所有使用者的代码（无悬垂指针，无 use-after-free），但原子表整体不可回收。

### 3.6.4 如果确实需要释放（练习 3.8）

原书练习 3.8 探讨了扩展接口：

```c
/* 练习 3.8: 原子释放扩展 */
extern void Atom_free (const char *str);  /* 释放单个原子 */
extern void Atom_reset(void);             /* 释放所有原子 */
```

实现 `Atom_free` 的难点在于：必须确保没有其他代码仍持有被释放的原子的指针（否则产生悬垂指针）。这是原子 "永不释放" 设计选择的核心原因。

## 3.7 Modern-C 改进（C11/C17/C23）

本章代码写于 C89/C90 时代。在现代 C 标准下，可以做以下改进：

### 3.7.1 `thread_local` — 线程局部存储

| 改进方向 | 原始做法 (C89) | 现代做法 (C11+) | 权衡 |
|----------|----------------|-----------------|------|
| **线程安全** | 全局 static | `thread_local` / 读写锁 / `_Atomic` | 内存开销 vs 锁竞争 |
| **哈希质量** | scatter 查表 | xxHash64 / WyHash | 更好的分布, 更多代码 |
| **变长结构** | `(char *)(p + 1)` | C99 FAM `char str[]` | 标准合规, 更安全 |
| **查找长度** | O(N) 全表扫描 | 指针反算 / 辅助表 | 额外存储 vs O(1) 查找 |
| **编译时生成** | 运行时静态数组 | C23 `constexpr` | 消除初始化开销 |
| **安全性** | 无类型检查 | `_Static_assert`, `restrict` | 纯增量改进 |

## 3.9 总结

Atoms 模块以不到 120 行 C 代码实现了一个完整的字符串驻留系统，是 CII 库核心设计哲学的典范：

1.  **接口极简**：4 个函数覆盖所有用例。`Atom_string` 和 `Atom_int` 是建立在 `Atom_new` 之上的便利层。
2.  **实现务实**：2048 个桶 + scatter 查表法，在简单性与效率之间取得令人满意的平衡。作者选择的不是 "理论上最优" 的方案，而是在实践中 "足够好" 的方案。
3.  **错误处理统一**：通过 `assert` 宏检查运行时错误（如无效参数），通过 `Except` 框架处理可恢复异常（如内存不足）。
4.  **变长结构体**：`p->str = (char *)(p + 1)` 是经典 C 语言技巧 —— 一块连续内存承载结构体和数据，减少内存碎片，提升缓存友好性。
5.  **永不释放**：原子的生命周期与程序相同。这一简化设计消除了悬垂指针和 use-after-free 的风险，且在实践中不会造成内存问题（C 程序中不同字符串的数量惊人地少）。
6.  **注意细节**：`(unsigned char)str[i]` 的强制转换、`LONG_MIN` 的专门处理、`do...while` 而非 `while` —— 每个细节都反映了作者对 C 语言陷阱的深刻理解。

### 延伸阅读

*   **LISP 与 SNOBOL4**：原子的命名源自 LISP；SNOBOL4 (Griswold 1972) 实现了几乎完全相同的字符串模型。
*   **lcc 编译器** (Fraser and Hanson 1995)：Atom 模块的前身。lcc 将所有标识符和常量的字符串存储在单一表中，且永不释放——这是 C 程序使用原子模式的一个成功先例。
*   **哈希理论**：Sedgewick (1990) 和 Knuth (1973b, *The Art of Computer Programming, Vol. 3: Sorting and Searching*) 提供了哈希函数的详细分析和设计指南。本章使用的哈希函数由 Hans Boehm 建议。
